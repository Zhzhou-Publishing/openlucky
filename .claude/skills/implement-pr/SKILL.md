# 按 PR 需求文档实现代码

## 动作

参数：`pr_no`（必填，整数，如 `21` 或 `021`）。

在项目根目录的 `pr/` 下查找需求文档，把内容当作需求规格，编写/修改代码实现它。

## 步骤

1. **定位需求文档**。在 `pr/` 目录下用 glob 模式 `<pr_no>.*.md` 和 `<pr_no>.*.txt` 查找文件（`*` 在 glob 里匹配文件名中段任意字符，对应你脑中的正则 `<pr_no>\..*\.(md|txt)`）。
2. **零填充回退**。如果第 1 步没命中，且 `pr_no` 不足 3 位，把它左侧补零到 3 位（如 `21` → `021`）再用同样的 glob 重试一次。
3. **结果分类处理**：
   - **0 个匹配**：报错告知用户「`pr/` 下找不到 PR 编号 X 的需求文档」，停止。
   - **1 个匹配**：直接读取该文件作为需求文档。
   - **多个匹配**（例如同时存在 `021.foo.md` 和 `021.bar.md`，或 `.md` 和 `.txt` 都有）：列出全部匹配文件让用户选定一个再继续。
4. **理解需求**。完整阅读需求文档，必要时再读相关代码上下文。如果需求里有歧义、缺失关键信息、或与现有代码冲突，**先向用户澄清，不要凭空猜测**。
5. **实现**。按需求修改/新增代码。遵守仓库已有的代码风格与目录约定。
6. **不提交**。改完后只汇报「改了哪些文件、做了什么」，把代码留在工作树，让用户后续走 `/commit-push` 等链路。

## 注意

- 不要修改需求文档本身（除非用户明确让你更新）。
- 不要执行 `git add`、`git commit`、`git tag`、`git push`。
- 如果需求文档中提到测试，按文档指示实现/运行测试；如果没提到，不要自作主张加大段单测。
- 实现过程中如果发现需求规格本身有问题（不可行、自相矛盾、风险大），停下来跟用户讨论，不要硬着头皮写。
